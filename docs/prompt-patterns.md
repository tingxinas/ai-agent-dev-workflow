# Prompt Patterns

本文件整理在 Agentic Development Workflow 中常用的提示词模式。目标不是追求复杂提示词，而是让 Agent 输出更稳定、更可验证。

## 1. 需求澄清模式

```text
请先不要写代码。请根据下面的需求，帮我澄清：
1. 目标用户
2. 需要解决的问题
3. 核心功能
4. 非目标范围
5. 验收标准
6. 风险点

需求：...
```

适用场景：需求还比较模糊，不适合直接开发。

## 2. 任务拆解模式

```text
请把这个项目拆成 5-7 个可执行任务。
每个任务包含：
- 目标
- 需要修改或创建的文件
- 验收标准
- 主要风险

要求：不要拆得太碎，优先按用户可见功能拆分。
```

适用场景：开始开发前制定计划。

## 3. 代码修改模式

```text
请基于现有代码风格修改，不要大规模重构。
请先说明你打算改哪些文件和为什么，再给出修改。
修改完成后，请列出验证步骤。
```

适用场景：已有项目中添加功能或修复问题。

## 4. Debug 分析模式

```text
下面是一个 Bug，请先分析可能原因，不要直接改代码。

现象：
复现步骤：
期望结果：
实际结果：
报错日志：
相关代码：

请输出：
1. 可能原因，按概率排序
2. 最小验证步骤
3. 推荐修复方案
4. 修复后的测试方式
```

适用场景：报错原因不明确时。

## 5. 代码审查模式

```text
请从以下角度审查这次修改：
1. 是否满足需求
2. 是否有边界条件遗漏
3. 是否有安全风险
4. 是否引入不必要复杂度
5. 是否需要补充测试或文档

请只指出重要问题，不要做泛泛评价。
```

适用场景：提交前审查。

## 6. 文档生成模式

```text
请根据当前项目内容生成 README 初稿，包括：
- 项目简介
- 解决的问题
- 使用方式
- 项目结构
- 核心流程
- 已知限制
- License

要求：语言真实，不夸大，不出现无法证明的数据。
```

适用场景：项目交付整理。

## 7. 发布检查模式

```text
请帮我检查这个项目是否适合公开发布到 GitHub。
重点检查：
1. 是否可能包含敏感信息
2. README 是否准确
3. 示例数据是否可公开
4. 文件结构是否清晰
5. 是否缺少 License 或运行说明
```

适用场景：开源或提交链接前。
