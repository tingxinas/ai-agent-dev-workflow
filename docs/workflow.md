# Agentic Development Workflow

本文档描述一套可复用的 AI Agent 辅助开发流程。它适合个人开发者、小团队、原型项目、内部工具和自动化脚本类项目。

## 1. Clarify：澄清问题

### 目标

让 Agent 先理解问题，而不是直接生成代码。

### 输入

- 一句话需求；
- 产品草图；
- 页面截图；
- 用户反馈；
- 报错日志；
- 已有代码仓库。

### 输出

- 项目目标；
- 用户对象；
- 核心功能；
- 非目标范围；
- 验收标准；
- 风险点。

### 推荐提示词

```text
请先不要写代码。请根据下面的需求，帮我澄清：
1. 项目目标
2. 目标用户
3. 核心功能
4. 非目标范围
5. 验收标准
6. 可能风险

需求：...
```

## 2. Plan：制定计划

### 目标

把需求拆成可以逐步完成的任务，减少返工。

### 输出

- 功能模块列表；
- 技术方案；
- 数据结构；
- 文件结构；
- 实现顺序；
- 测试路径。

### 推荐提示词

```text
请把这个需求拆成 5-7 个可执行任务。每个任务需要包含：目标、涉及文件、验收标准和风险点。不要把任务拆得太碎。
```

## 3. Build：实现功能

### 适合 Agent 做的事

- 生成页面或组件初稿；
- 编写工具函数；
- 重构重复逻辑；
- 根据已有代码风格补全功能；
- 增加错误处理；
- 生成测试样例；
- 补充文档。

### 不建议完全交给 Agent 的事

- 关键业务规则最终判断；
- 权限、安全、支付相关逻辑；
- 未经确认的大规模重构；
- 删除或迁移重要数据；
- 对外发送消息或执行不可逆操作。

## 4. Debug：排查问题

### 输入格式

```text
现象：

复现步骤：
1.
2.
3.

期望结果：

实际结果：

报错日志：

相关代码：
```

### Agent 输出要求

- 先列出 3-5 个可能原因；
- 按概率和影响排序；
- 给出最小验证步骤；
- 再给修复建议；
- 避免一次性改太多地方。

## 5. Review：人工审核

### 检查清单

- 需求是否被正确实现；
- 是否有多余功能；
- 是否引入不必要依赖；
- 是否存在密钥、Token 或隐私信息；
- 是否有明显性能问题；
- 是否影响已有功能；
- 文档是否与实际行为一致。

## 6. Ship：交付发布

### 最小交付内容

- README；
- 运行方式；
- 主要功能说明；
- 已知限制；
- 测试或验证记录；
- 变更记录。

### 发布前确认

- 项目可以从零拉取后运行；
- 敏感文件已被忽略；
- 截图和示例数据可以公开；
- License 明确；
- README 中没有夸大描述。

## 7. Document：沉淀经验

每完成一个项目，建议记录：

- 哪些任务适合 Agent；
- 哪些任务必须人工判断；
- 哪类提示词效果最好；
- 哪些错误容易反复出现；
- 下次可以复用的模板。

## 总结

Agentic Development 的核心不是“让 AI 全自动写完项目”，而是让开发过程更结构化：先理解，再计划，小步执行，持续验证，最后沉淀。
